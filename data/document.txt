SECTION 32: SECURITY PRACTICES

32.1 Infrastructure Security: DataSphere's infrastructure operates within SOC 2 Type II certified environments. All compute, storage, and networking components are logically isolated using industry-standard virtualization and containerization technologies. We employ defense-in-depth principles with multiple layers of security controls.

32.2 Data Encryption: All data stored in DataSphere's systems is encrypted at rest using AES-256. All data transmitted to and from our APIs is encrypted using TLS 1.3. Encryption keys are managed using hardware security modules (HSMs) and rotated regularly.

32.3 Access Control: Access to DataSphere's production systems is strictly controlled. All internal access requires multi-factor authentication. Access is granted on a need-to-know basis using role-based access control (RBAC). All privileged access sessions are logged and monitored.

32.4 Vulnerability Management: DataSphere conducts regular security assessments including:
- Continuous automated vulnerability scanning
- Quarterly penetration testing by independent security firms
- Annual third-party security audits
- Bug bounty program (details at security.datasphere.ai/bounty)

32.5 Incident Response: DataSphere maintains a formal incident response plan with defined procedures for detecting, containing, eradicating, and recovering from security incidents. In the event of a security incident affecting your data, we will notify you in accordance with applicable laws and our DPA.

- Acknowledging receipt of your report within 2 business days
- Providing an initial assessment within 5 business days
- Working collaboratively to understand and address the issue
- Not pursuing legal action against researchers who follow our responsible disclosure guidelines

- Model deployments: 2

Professional Plan:

- 100 requests per second (RPS)

- Concurrent training jobs: 5

Business Plan:
- 20,000,000 API calls per month

- Concurrent training jobs: 20

Enterprise Plan:
- Unlimited API calls (subject to fair use policy)
- Custom RPS limits
- Custom payload size limits
- Unlimited concurrent training jobs
- Unlimited model deployments

36.2 Rate Limit Headers: API responses include headers indicating your current rate limit status:
- X-RateLimit-Limit: Your total rate limit
- X-RateLimit-Remaining: Remaining calls in current window
- X-RateLimit-Reset: Unix timestamp when the rate limit resets

36.3 Exceeding Rate Limits: Requests that exceed your rate limits will receive a 429 Too Many Requests response. Implement exponential backoff with jitter in your API clients to handle rate limiting gracefully.
